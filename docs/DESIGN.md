## Problem Statement

Crypto markets are driven by information asymmetry. When a major event happens (tweet, news, hack), some wallets trade within seconds while most react minutes or hours later. No existing tool maps the full causal chain: **event → who engaged → which wallets moved → price impact**. Arkham and Nansen are wallet-first (start from a wallet, see what it did). This tool is event-first (start from an event, trace who moved first).

## What Makes This Cool

A 3D force graph visualization shows information propagating through a network of wallets in real-time — like watching a shockwave ripple through the market. The center node is the event (a tweet, a headline). Rings of wallets light up by reaction time. Animated particles flow along edges showing money movement. Then the punchline: the tool identifies wallets that **consistently** front-run the crowd — an "alpha leak detector."

The demo moment: judge types an event → graph builds live on screen → a specific wallet is revealed as a repeat fast-mover across multiple events.

3. The 3D force graph visualization is the hook; wallet forensics is the substance
4. 4 hours is enough with react-force-graph-3d (drop-in component) and a focused scope
5. Demo is live/interactive — judge types events, system responds in real-time

## Approaches Considered

### Approach A: Scripted Reveal (NOT CHOSEN)
Fully pre-computed, 2-3 curated events, maximum polish. Safest but not interactive. Judges can't try their own events.

### Approach B: Interactive Forensics (CHOSEN)
Fully live tool. Judge types an event description → Claude maps it to a timestamp/token → HL API pulls trades in that window → graph renders live → wallet analysis runs. Most impressive, highest risk.

### Approach C: Hybrid
Scripted core + live wallet search. Reliable but less impressive than full interactivity.

## Recommended Approach

**Approach B: Interactive Forensics** — with mitigation strategies for the live demo risk.

### Hyperliquid API Specifics

HL Info API (POST `https://api.hyperliquid.xyz/info`):
- `userFills`: trades by wallet address — returns `{coin, px, sz, side, time, fee, oid}` with millisecond timestamps
- `clearinghouseState`: current positions per wallet
- `allMids`: current mid prices for all tokens
- `candleSnapshot`: OHLCV candles for price data

**Key constraint**: `userFills` is per-wallet, not per-token. To get "all trades on ETH in a window," we need to either (a) query known active wallets, or (b) use the `recentTrades` websocket/REST endpoint which returns all fills for a coin. Use `recentTrades` for the initial scan, then `userFills` for deep-diving specific wallets.

**Rate limits**: No documented hard limits for read endpoints. Use reasonable batching (1-2 req/sec).

**Historical depth**: HL launched mainnet Nov 2023. All events must be post-Nov 2023. LUNA depeg (May 2022) is NOT available — use recent events only (2024-2026).

**CRITICAL — `recentTrades` endpoint must be verified in the first 15 minutes of the hackathon.** If no historical time-range query exists, the fallback is: use `userFills` for a pre-curated list of ~50 known active wallets (whale addresses from Arkham/HL leaderboard). This is less comprehensive but still produces a compelling graph.

**P&L calculation**: Entry price = wallet's first fill price in the window. Exit price = mid price at T+15min (from `candleSnapshot`). P&L = (exit - entry) / entry × direction_sign. Cap profit_multiple at 10 to prevent outlier distortion.

### Claude Prompt Schema

```typescript
interface EventQuery {
  userInput: string;  // "Elon tweets about DOGE, March 2025"
}

interface ParsedEvent {
  token: string;       // "DOGE" — must map to HL coin name
  timestamp: string;   // ISO 8601, e.g. "2025-03-15T14:30:00Z"
  direction: "pump" | "dump" | "volatile";
  confidence: number;  // 0-1, reject below 0.5
  reasoning: string;   // one-line explanation
}
```

Claude system prompt: "You are a crypto event analyst. Given a user's description of a crypto market event, extract the primary token affected (use the ticker symbol as listed on Hyperliquid), the approximate UTC timestamp, and whether the event caused a pump, dump, or volatility spike. Only reference events after November 2023 (Hyperliquid's launch). If the event is ambiguous or you're unsure of the timestamp, set confidence below 0.5."

### Data Model

```typescript
interface GraphNode {
  id: string;           // wallet address or "event-{hash}"
  type: "event" | "wallet";
  group: 0 | 1 | 2 | 3; // 0=event, 1=<60s, 2=1-5min, 3=5-30min
  label: string;        // truncated address or event title
  size: number;         // trade volume (normalized 1-20)
  reactionTime?: number; // seconds after event
  pnl?: number;         // profit/loss on this trade
  entity?: string;      // Arkham label if available
}

interface GraphLink {
  source: string;       // node id
  target: string;       // node id
  value: number;        // trade size (drives particle speed)
  direction: "buy" | "sell";
  color: string;        // green=buy, red=sell
}
```

### Data Pipeline (per event query)

1. **User types event**: "ETH flash crash Jan 2025" or "HYPE token launch pump"
2. **Claude API** → returns `ParsedEvent` with token, timestamp, direction, confidence. Reject if confidence < 0.5.
3. **Price data** (HL `candleSnapshot`) → confirms price spike in ±30 min window, identifies exact peak/trough timestamp
4. **Hyperliquid API** (`recentTrades` for the coin) → pulls all fills in the ±30 min window around event. If `recentTrades` doesn't support historical range queries, fall back to `userFills` for top 50 known whale wallets.
5. **Analysis engine** → groups fills by wallet address, computes per-wallet: first trade timestamp, net direction, volume, estimated P&L
6. **Graph builder** → constructs `{nodes: GraphNode[], links: GraphLink[]}`:
   - Center node: the event (type "event", group 0)
   - Ring 1 (red/hot, group 1): wallets that traded within 60 seconds
   - Ring 2 (orange, group 2): 1-5 minutes
   - Ring 3 (blue/cool, group 3): 5-30 minutes
   - Links connect event → each wallet, with particles animated by volume
7. **Wallet scoring** → for wallets appearing across multiple cached events, compute alpha score

### Live Demo Risk Mitigation

**Problem**: LLM + API calls = 10-30 seconds latency on stage. Judges get bored.

**Mitigations**:
1. **Pre-cached fallback**: P4 pre-computes results for 5-10 known events. If the live pipeline is slow, the app checks the cache first. Known events return instantly. Novel events go through the full pipeline.
2. **Progressive rendering**: Since HL's `recentTrades` returns a single response (not streaming), "progressive" means: show the event node + price chart immediately after Claude responds, THEN add wallet nodes in batches (e.g., 10 at a time with 100ms delay) after the HL data arrives. The staggered node appearance creates an animation effect.
3. **Loading UX**: While waiting, show: event parsing result, price chart with impact highlighted, "Scanning X trades..." counter. The wait becomes part of the experience.
4. **Demo script**: Start with a cached event to show the full experience instantly. THEN say "but let me show you it works live" and type a novel event. If the live one fails, the cached demo already proved the concept.

### Wallet Alpha Score Formula

```
alpha_score(wallet) = Σ over events [
  (1 / reaction_time_seconds) × profit_multiple × event_significance
]
```

Where:
- `reaction_time_seconds`: time between event timestamp and wallet's first trade (clamped to min 1s to avoid division by zero)
- `profit_multiple`: return on the trade (1.5 = 50% gain, capped at 10 to avoid outlier distortion)
- `event_significance`: absolute percentage price change in the 15-min window (e.g., 5% move = 0.05). NOT normalized — raw magnitude. Bigger events weigh more.

Higher score = consistently fast + consistently profitable across events.

### Cross-Event State

Alpha scores require data across multiple events. Storage: an in-memory `Map<walletAddress, WalletProfile>` built during P4's pre-caching phase. When a live query runs, new wallet appearances are added to the map. No database needed — the map resets on server restart, which is fine for a hackathon demo.

### Error States

| Failure | User sees | Behavior |
|---------|-----------|----------|
| Claude confidence < 0.5 | "I couldn't pinpoint that event. Try being more specific — include the token name and approximate date." | No API calls made |
| Token not on HL | "That token isn't traded on Hyperliquid. Try BTC, ETH, SOL, DOGE, or HYPE." | Show list of available tokens |
| HL returns zero trades | "No trades found in that window. The event may not have impacted Hyperliquid traders." | Show empty graph with event node only |
| HL API timeout/error | "Hyperliquid API is slow — loading from cache..." | Fall through to cached results if available |
| Wallet appears in < 2 events | Not scored in alpha leaderboard | Only wallets with 2+ event appearances get an alpha score |

## Open Questions

1. **Which HL tokens have enough historical events with clear social triggers?** BTC and ETH perps are safest. DOGE, SOL, HYPE are riskier but more interesting.
2. **HL API rate limits for historical trade data** — need to verify bulk pull is possible without hitting limits.
3. **How far back does HL's trade history API go?** Need at least 6 months for enough events.
4. **Can we get tweet timestamps reliably?** Twitter/X API is rate-limited. May need to use pre-known event timestamps.
5. **What if no wallets show consistent fast-moving behavior?** CRITICAL — validate this in the first 30 minutes of the hackathon. P4 should pull trade data for 2-3 known events and check if any wallet addresses appear as fast-movers in multiple events. If no overlap exists, pivot the narrative from "alpha leak detector" to "information propagation visualizer" (still compelling, just different punchline). The graph visualization works either way.
